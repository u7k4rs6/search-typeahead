# Performance Report

All figures from a local run on the synthetic 214,816-query Zipfian dataset (exponent 0.9). Load generated by `npm run loadtest` (5,000 operations: 3,759 suggest, 1,241 search; concurrency 25). Ring balance measured by `npm run ringdist` over the full key space.

## Latency

Latency is reported at two points: end-to-end round-trip measured client-side (includes HTTP parsing, the loopback TCP stack, and JSON serialization), and suggestion-handler compute measured server-side (the trie and cache lookup only, timed with `performance.now()` for fractional-millisecond resolution).

| Percentile | Client end-to-end | Server lookup |
|---|---|---|
| p50 | 2.73 ms | 0.002 ms |
| p95 | 10.07 ms | 0.009 ms |
| p99 | 16.59 ms | 0.019 ms |

The core suggestion lookup runs in single-digit microseconds at the median (about 2) and under 20 microseconds at p99. The gap to the client figure is HTTP and framework overhead, not the algorithm.

## Cache hit rate

3,584 hits against 175 misses over 3,759 suggestion requests, a **95.3% hit rate**. The rate reflects workload locality: the load test reuses a pool of popular prefixes, as real search traffic does, and would fall under a more uniform or adversarial query mix. Only basic-mode requests touch the cache; enhanced-mode requests are computed fresh and are excluded from the hit and miss counters.

## Write reduction through batching

Search submissions are buffered and aggregated, with duplicate queries collapsed into a single increment, before flushing to the store.

| Metric | Count |
|---|---|
| Searches submitted (enqueued) | 1,241 |
| Store writes (flushed) | 300 |
| Writes avoided | 941 |
| Reduction | **75.8%** |

Synchronous writing would have issued one store write per search (1,241). Batching with aggregation reduced that to 300. The trade-off: counts buffered but not yet flushed are lost if the process crashes before a flush, an accepted durability-for-throughput choice. A write-ahead log is the named mitigation if at-least-once delivery is required.

## Consistent-hash ring balance

The ring uses 5 physical nodes with 150 virtual nodes each. Initial measurement exposed a 2.04x load imbalance, traced to FNV-1a's weak avalanche: the near-identical virtual-node labels (`node-0#vnode#0`, `#1`, and so on) hashed to clustered ring positions, collapsing 150 virtual nodes to roughly 3 effective positions per node. Passing the FNV-1a output through a MurmurHash3 fmix32 finalizer restored uniform placement.

| Node | Before (FNV-1a) | After (FNV-1a + fmix32) |
|---|---|---|
| node-0 | 40.79% | 20.16% |
| node-1 | 20.43% | 17.74% |
| node-2 | 12.54% | 19.34% |
| node-3 | 8.14% | 20.81% |
| node-4 | 18.10% | 21.95% |
| Std dev (% of mean) | 56.2% | 7.1% |
| Imbalance (max / mean) | 2.04x | 1.097x |

Theory for 150 virtual nodes per node predicts a load standard deviation near 1 over the square root of 150, about 8% of the mean, and a max-to-mean ratio near 1.1x. The post-fix measurement (7.1% and 1.097x) matches, confirming the ring distributes the full key space as consistent hashing should. The live cache over the load test's hot prefix pool also showed no dominant node after the fix.

## Trending: basic versus enhanced

The same `/api/suggest` endpoint serves two rankings. Basic ranks by all-time count. Enhanced re-ranks a top-100 candidate pool by an exponential time-decayed score, so a recently surging query rises even when its all-time count is lower.

For the prefix `ip` after a burst of searches for "iphone 15 pro max":

- **Basic top-10** led by "iphone 14 price", "ipad", "iphone case"; "iphone 15 pro max" does not appear, since its all-time count sits outside the prefix top 10.
- **Enhanced top-5** surfaces "iphone 15 pro max" to first, ahead of the all-time leaders.

This demonstrates recency overriding raw popularity without permanently over-ranking the burst: the decayed score falls back toward the all-time order as the surge ages.
